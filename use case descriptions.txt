12.2 Preliminary Use Case Description
12.2.1 Admin 
Name: Admin login
ID: 1
Stakeholders and Goals: Admin securely accesses the admin dashboard to manage the system.
Description: Admin submits email/password; backend verifies credentials, issues a session token, and returns the admin redirect.
Actors: Admin
Pre-condition(s): Admin account exists with correct credentials.
Post-condition(s): Session created; token/expiry returned; redirect target /dashboard/admin provided.
Trigger(s): Admin clicks Log In and submits credentials.
Normal Flow(s):

Admin clicks Log In and submits credentials.
UI calls /api/auth/login with credentials.
Server checks password, ensures role admin, fetches admin profile (name/phone).
Server issues session token + expiry and returns user payload with redirect.


Sub-Flow(s): None.
Alternative Flow(s):
A1: Invalid credentials → 401.
A2: Unsupported role → 403/404.
A3: Session creation error → 500.













Name: Admin logout
ID: 2
Stakeholders and Goals: Admin ends session so the dashboard stays secure.
Description: Admin triggers logout; backend revokes the session token.
Actors: Admin
Pre-condition(s): Admin authenticated; presents session token.
Post-condition(s): Session revoked; client token cleared; further requests require login.
Trigger(s): Admin clicks Logout.
Normal Flow(s):

Admin clicks Logout.
UI sends logout with bearer/body token to /api/auth/logout.
Server revokes session via session_service.revoke_session.
UI clears token and returns to login.
Sub-Flow(s): None.
Alternative Flow(s):
A1: Missing/invalid token → 400/404; UI treats admin as logged out.

















Name: View admin dashboard
ID: 3
Stakeholders and Goals: Admin spots performance issues and plans improvements.
Description: Admin opens the dashboard; backend returns CSAT summary, trend, and recent feedback.
Actors: Admin

Trigger(s): Admin opens the admin dashboard.

A1: No CSAT data → empty summary/trend returned.

Name: Escalate to live agent (ticketized, customer or guest)
ID: 14
Stakeholders and Goals: Customer gets human help when chatbot cannot assist.
Description: During chatbot handoff, backend creates a chat session with a ticket number, captures customer_id if logged in (or creates a guest ticket), and queues the session for agents.
Actors: Customer (or guest), Support Agent
Pre-condition(s): Chatbot escalates or customer requests human help.
Post-condition(s): Chat session/ticket created with customer identity; queue position available; messages routed to agents.
Trigger(s): Handoff event from chatbot/Rasa.
Normal Flow(s):

User clicks “Talk to Human” quick-reply button or type the request
Handoff event from chatbot/Rasa.
For logged-in users, handoff passes customer_id; create_session_from_rasa stores it and generates a ticket number.
System appends the initial customer message and places session in pending queue.
Agents claim; customer sees queue status via queue_status; chat proceeds.
Sub-Flow(s): S1 (Guest handoff): create_guest_ticket creates guest user + session, generates ticket number, seeds first message, and queues it.
Alternative Flow(s):
A1:No open session for sender on append → 404; 
A2:customer message before claim → 409 until an agent claims.
